When a Standard User Spawned an Administrator Command Prompt

An anonymized engineering case study on DLL search order hijacking, unsafe library loading, privilege misuse, Windows internals, detection engineering, and secure application design.

Privilege misuse, arbitrary code execution in a trusted process, defense evasion, and potential persistence.

Executive Summary

During quality assurance testing of a commercial Windows application, I encountered a vulnerability in which the application loaded an attacker-controlled DLL from its installation directory.

Under normal conditions, a non-administrative user launched the application and it behaved as expected. However, when a crafted library with a name expected by the application was placed in the application directory, Windows loaded that library during startup. The DLL executed inside the trusted application process and resulted in an elevated command prompt being presented to the standard user.

Important distinction:

DLL hijacking explains how attacker-controlled code entered the process. The elevated outcome depended on the application's privileged execution model, service interaction, helper process, scheduled task, inherited token, or another trusted component. DLL loading alone does not automatically elevate privileges.

The issue was reproduced in a controlled environment, documented, escalated to engineering, and remediated. This case study focuses on the defensive lessons: how to identify the root cause, detect exploitation, and prevent similar weaknesses in Windows applications.

The Story

Everything looked normal.

A standard Windows user signed in and launched the application. There was no UAC prompt, no administrator token, and no unusual child process. Application functions and permissions behaved as designed.

Then one variable changed.

A DLL was placed in the application's directory. The user launched the same executable again. This time, an administrator command prompt appeared.

The executable had not changed. The user had not changed. The application had trusted different code.

The investigation shifted from “why did Windows elevate this user?” to a more useful question: “what library did the application load, from where, and under which security context?”

How DLL Search Order Hijacking Works

Windows applications use Dynamic Link Libraries to reuse code. A program may resolve DLL dependencies during startup through its import table or load them dynamically at runtime through functions such as LoadLibrary and LoadLibraryEx.

When an application specifies only a DLL filename rather than a trusted absolute path, the Windows loader searches a sequence of locations. The exact order varies according to application configuration, Windows version, Safe DLL Search Mode, KnownDLLs, packaged-app rules, redirection, manifests, and the flags used.

Application.exe │ ├── Requests "example.dll" │ ▼ Windows Loader │ ├── Checks loader rules and KnownDLLs ├── Checks application-related locations ├── Checks Windows system directories ├── Checks current directory / configured search paths ▼ First matching DLL is loaded
The weakness:

A standard user must be able to influence a directory searched before the legitimate library, or replace a DLL in a location the application trusts. If the application directory is properly protected, simply being a standard user should not be enough to drop a DLL there.

The Investigation

1. Establish the baseline

The first step was to capture normal behavior: process tree, loaded modules, file access, token integrity, application services, and child processes. A clean baseline made the altered launch easy to compare.

2. Observe file and image loading

Process Monitor and EDR image-load telemetry can reveal the exact DLL path selected by the loader. Useful filters include the application process name, Load Image operations, and unsuccessful attempts such as NAME NOT FOUND.

3. Compare security contexts

The application process, helper process, service, and resulting shell should be compared for: user SID, integrity level, elevation type, parent-child relationship, token source, and session.

4. Validate write permissions

Access control on the installation directory is critical. A weak ACL allowing standard users to create or replace DLLs can transform unsafe loading behavior into a practical local privilege escalation path.

5. Confirm causality

Removing the crafted DLL should restore normal behavior. Renaming it should prevent loading. Reintroducing it should reproduce the result. This controlled repetition separates causation from coincidence.

Root Cause Analysis

The vulnerability required a chain of trust failures rather than one isolated defect.

ConditionWhy it mattered
Unsafe DLL resolutionThe application requested a library without restricting the loader to a trusted location.
User-controlled placementA standard user could place or influence a DLL in a searched directory.
Trusted execution contextThe DLL executed within a process or component with access beyond the user's normal privileges.
Dangerous child-process behaviorThe compromised process could create an elevated shell or invoke a privileged helper.

Unsafe pattern

HMODULE module =
    LoadLibraryW(L"example.dll");

The loader must search for the library using configured search rules.

Safer pattern

HMODULE module = LoadLibraryExW(
    L"example.dll",
    NULL,
    LOAD_LIBRARY_SEARCH_SYSTEM32
);

The search is restricted to a trusted location when the dependency belongs in System32.

For application-owned libraries, the safer design is to build a canonical absolute path from a protected installation directory and use appropriate LoadLibraryEx flags. Developers should also consider SetDefaultDllDirectories and AddDllDirectory.

Why Did an Administrator Command Prompt Appear?

A malicious DLL normally executes with the privileges of the process that loads it. Therefore, the elevated result indicates that the vulnerable application interacted with an already elevated component or possessed a privileged execution path.

Possible architectures include:

  • A privileged Windows service loading or acting on data controlled by the user-facing application.
  • An auto-elevated helper executable or broker process.
  • A scheduled task configured to run with highest privileges.
  • An elevated updater, maintenance component, or service control channel.
  • A privileged process that loaded the DLL directly from an unsafe location.
Key analytical point:

The DLL hijack was the code-execution mechanism. The privilege boundary failure was the application architecture that allowed that code to execute or trigger actions in an elevated context.

Attack Flow

Standard User │ ├── Gains write access to searched directory │ ├── Places crafted DLL using expected filename │ ▼ Launches Application.exe │ ▼ Windows loader resolves dependency │ ├── Attacker-controlled DLL found first ▼ DLL executes inside trusted process │ ├── Invokes privileged helper / service path ▼ Elevated cmd.exe is created

Detection Engineering

Reliable detection should correlate library loading with path trust, signing status, file creation, process ancestry, and privilege context. Alerting on every non-System32 DLL will produce excessive noise.

TelemetryDetection value
Sysmon Event ID 7Records image loads when enabled; useful for unexpected DLL paths and signatures.
Sysmon Event ID 11Shows creation of a DLL shortly before application launch.
Sysmon Event ID 1Correlates the compromised application with suspicious child processes such as cmd.exe.
EDR image-load eventsOften provide signer, prevalence, hash, process, and path context.
File ACL auditingIdentifies weak permissions in privileged application directories.

High-value behavioral sequence

New or modified DLL + User-writable / unexpected path + Loaded by trusted application + Unsigned or low-prevalence module + Elevated shell or unusual child process

Splunk Detection Examples

Unexpected DLL loaded by a sensitive application

index=sysmon EventCode=7
| eval dll_path=lower(ImageLoaded)
| where like(dll_path, "%.dll")
| where NOT like(dll_path, "c:\\windows\\system32\\%")
| stats count min(_time) as first_seen max(_time) as last_seen
        values(Signed) as signed
        values(Signature) as signer
        by host Image ImageLoaded ProcessGuid
| where signed!="true" OR isnull(signed)

DLL creation followed by application launch

index=sysmon (EventCode=11 TargetFilename="*.dll") OR EventCode=1
| eval correlation_path=coalesce(TargetFilename, Image)
| transaction host maxspan=5m
| search EventCode=11 EventCode=1
| table _time host User Image ParentImage CommandLine TargetFilename

Trusted application spawning an elevated shell

index=sysmon EventCode=1
(Image="*\\cmd.exe" OR Image="*\\powershell.exe")
| where match(lower(ParentImage), "\\\\program files\\\\")
| table _time host User IntegrityLevel ParentImage Image CommandLine

Microsoft Sentinel / Defender XDR

Unsigned DLL from an unexpected application path

DeviceImageLoadEvents
| where FileName endswith ".dll"
| where FolderPath !startswith @"C:\Windows\System32\"
| where IsSigned == false or isempty(Signer)
| project Timestamp, DeviceName, InitiatingProcessFileName,
          InitiatingProcessFolderPath, FileName, FolderPath,
          SHA1, IsSigned, Signer
| order by Timestamp desc

DLL creation followed by load and suspicious child process

let lookback = 10m;
let dllWrites =
    DeviceFileEvents
    | where ActionType in ("FileCreated", "FileModified")
    | where FileName endswith ".dll"
    | project DeviceId, DLLTime=Timestamp, DLLName=FileName,
              DLLPath=FolderPath, SHA1;
let dllLoads =
    DeviceImageLoadEvents
    | where FileName endswith ".dll"
    | project DeviceId, LoadTime=Timestamp,
              InitiatingProcessId, InitiatingProcessFileName,
              LoadedDLL=FileName, LoadedPath=FolderPath;
dllWrites
| join kind=inner dllLoads on DeviceId
| where LoadTime between (DLLTime .. DLLTime + lookback)
| where DLLName == LoadedDLL
| project DLLTime, LoadTime, DeviceId, InitiatingProcessFileName,
          LoadedDLL, LoadedPath, SHA1

Application spawning a shell

DeviceProcessEvents
| where FileName in~ ("cmd.exe", "powershell.exe", "pwsh.exe")
| where InitiatingProcessFolderPath startswith @"C:\Program Files\"
| project Timestamp, DeviceName, AccountName, FileName,
          ProcessCommandLine, InitiatingProcessFileName,
          InitiatingProcessFolderPath, InitiatingProcessIntegrityLevel

Sigma Detection Concept

title: Suspicious Shell Spawned by Installed Application
id: 2e32f4b8-f487-4c78-a7f1-4da6cfdc2410
status: experimental
description: Detects command shells launched by applications installed under Program Files.
logsource:
  category: process_creation
  product: windows
detection:
  selection_parent:
    ParentImage|startswith:
      - 'C:\Program Files\'
      - 'C:\Program Files (x86)\'
  selection_child:
    Image|endswith:
      - '\cmd.exe'
      - '\powershell.exe'
      - '\pwsh.exe'
  condition: selection_parent and selection_child
falsepositives:
  - Legitimate administrative or troubleshooting functions
level: medium
tags:
  - attack.privilege-escalation
  - attack.defense-evasion
  - attack.t1574.001

This rule is intentionally broad and should be tuned to known application behavior, signer reputation, integrity level, command line, and module-load evidence.

Prevention and Secure Design

For developers

  • Use absolute, canonical paths for private application DLLs.
  • Use LoadLibraryEx with restrictive search flags.
  • Call SetDefaultDllDirectories where compatible.
  • Use AddDllDirectory only for trusted protected locations.
  • Validate digital signatures for sensitive plug-ins or modules.
  • Do not allow elevated services to load dependencies from user-controlled paths.

For administrators

  • Ensure Program Files and service directories are not writable by standard users.
  • Deploy WDAC or AppLocker where appropriate.
  • Monitor ACL drift on privileged application directories.
  • Use EDR controls to block untrusted modules in sensitive processes.
  • Remove unnecessary auto-elevation and privileged helper behavior.

Example ACL review

icacls "C:\Program Files\Vendor\Application"

Standard users should generally have read and execute permissions, not create, modify, or delete rights in a privileged application directory.

Incident Response Workflow

  1. Isolate the endpoint if exploitation is suspected.
  2. Preserve the suspicious DLL, hashes, timestamps, signer data, and original path.
  3. Capture the complete process tree and process tokens.
  4. Identify all systems containing the same DLL hash or filename-path combination.
  5. Review file creation events to determine how the DLL arrived.
  6. Check whether the application directory ACL was modified.
  7. Review service, scheduled-task, updater, and helper-process interactions.
  8. Remove the malicious library and patch or reconfigure the vulnerable application.
  9. Hunt for persistence, credential access, lateral movement, and follow-on execution.

Lessons Learned

At first glance, the issue looked like a Windows privilege escalation flaw. The deeper investigation showed that Windows was following its loader rules. The real weakness was the application's trust model.

The application trusted a filename without sufficiently constraining its source. A privileged component then allowed code reached through that trust decision to cross a security boundary.

Do not investigate only which executable ran. Investigate every library, plug-in, helper, and service that the executable trusted.

That lesson continues to shape how I approach application security testing, endpoint telemetry, and threat hunting. A signed executable can still become the vehicle for malicious code when it loads an untrusted module.
