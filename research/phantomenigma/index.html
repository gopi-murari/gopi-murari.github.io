Research note

This article is an independent defensive analysis based primarily on ANY.RUN's PhantomEnigma investigation. It does not claim discovery of the original campaign. Indicators are defanged, and all detection content should be validated and tuned before production use.

Executive Summary

Security teams are trained to distrust spoofed domains, failed authentication checks, and unfamiliar websites. PhantomEnigma demonstrates the opposite problem: an attack can appear credible because the infrastructure is genuinely trusted.

The operation abused compromised Brazilian government mailboxes and web portals as part of its delivery chain. In this situation, SPF, DKIM, and DMARC may all pass—not because the message is safe, but because the attacker is sending through infrastructure that has already been compromised.

The malware chain combines a Delphi-compiled Inno Setup installer with a patched Electron or Node.js application. A malicious resources\app\index.js performs system discovery, persistence, C2 beaconing, JavaScript execution, and second-stage payload delivery.

The most important defensive lesson is that domains, IP addresses, and lure names rotate quickly. The build chain and behavioral relationships are harder to change. ANY.RUN identified 231 sandbox sessions sharing the recurring Node.js and Inno Setup combination, while the known C2 seed set covered only a minority of that wider cluster.

Core detection principle

Do not hunt only for where the malware connects. Hunt for how it is built, launched, persisted, and used to execute follow-on activity.

Why This Case Matters

PhantomEnigma did not simply bypass trust controls. It weaponized the conditions that make users and security products trust a message.

Email authentication answers a narrow question: was the sender authorized to use the sending domain? It does not answer whether the mailbox, server, or domain itself has been compromised.

This creates a visibility gap. A recipient may see a legitimate government domain. The secure email gateway may see passing authentication. The browser may see HTTPS. The SOC may initially see unrelated alerts: a downloaded installer, an Electron application, a Run-key change, and periodic web traffic.

The incident becomes clear only when those signals are correlated.

1. Trusted phishing delivery

The lure impersonates official Polícia Civil or “Procuração Digital” communications. Some activity also used QR-code documents and ClickFix-style social engineering. The delivery link may point to a compromised government host or a police-themed typosquatted domain.

The key distinction is that compromised government systems were used as delivery infrastructure. They should not automatically be treated as confirmed campaign victims or attacker-owned infrastructure.

2. Silent installer execution

The victim launches an installer such as Procuracao_Digital.exe. The Delphi/Inno package silently extracts an application into an Inno temporary directory:

%TEMP%\is-*.tmp\<RandomName>Application\

Observed naming patterns included random combinations such as DistroniceApplication, SmartSuiteifyUltraWare, and similar synthetic product names.

3. Electron application abuse

The extracted application appears to be a normal Electron program. Observed executable names included App.exe, Boost Note.exe, and later Grape.exe.

Electron loads JavaScript from the application resources directory. PhantomEnigma abuses that design by placing malicious logic inside:

resources\app\index.js

The executable may look ordinary. The process relationship is not.

4. Discovery and victim registration

The backdoor collects host details including the computer name, username, campaign tag, and a persistent eight-character machine identifier stored in the user profile.

5. Persistence and beaconing

The malware can establish startup persistence through a Windows Run key or Electron's setLoginItemSettings() functionality. It then checks for commands at a regular interval—approximately every 180 seconds in the newly analyzed generation.

6. Modular task execution

The server can return JavaScript for in-process execution through eval(), or send Base64-encoded files that are written to disk and launched. This allows the operator to choose a stealer, loader, remote-management tool, or other payload after the initial compromise.

Inside the Backdoor

ANY.RUN analysts recovered and statically deobfuscated a large index.js file without executing the code through Node.js or eval(). The analysis identified multiple string-obfuscation layers and a distinct beacon generation.

Detection implication

An Electron application that creates persistence, performs host discovery, launches command interpreters, or downloads executables should be investigated even when its filename resembles a legitimate desktop application.

Two beacon generations

The campaign used at least two communication formats:

GET /laravel.php?api=api&hash=<base64>&message=PT1n<base64>

POST /nbw/
Content-Type: application/json

The protocol changed, but the surrounding build chain remained recognizable: Delphi/Inno delivery, patched Electron, modular JavaScript or executable tasking, and startup persistence.

Infrastructure: What to Block and What Not to Paint

The operation used multiple infrastructure tiers:

  • Compromised delivery surface: legitimate Brazilian government and public-sector portals.
  • Delivery and C2 domains: police-themed typosquats and randomly generated-looking .com domains.
  • Proxy layer: Cloudflare-fronted domains.
  • Origin and beaconing tiers: separate hosting and bare-IP infrastructure.

Cloudflare co-hosting is not attribution. Shared edge IP space hosts large numbers of unrelated sites. Likewise, legitimate government hosts used for delivery should not be broadly blocked without considering business impact.

Attribution discipline

Incident Response Priorities

Immediate containment

  1. Isolate affected endpoints.
  2. Preserve the original email, headers, URLs, QR destination, and downloaded files.
  3. Reset exposed credentials and revoke active sessions where compromise is suspected.
  4. Block confirmed attacker-controlled domains and IPs.
  5. Avoid broadly blocking legitimate government domains used as compromised delivery hosts.
  6. Terminate malicious Electron, PowerShell, shell, and second-stage processes.

Enterprise scoping

  • App.exe or renamed Electron binaries under is-*.tmp
  • resources\app\index.js created by an installer
  • Run-key values referencing Temp or AppData
  • POST /nbw/ and GET /laravel.php?api=api
  • Electron applications launching PowerShell, CMD, WMI, registry tools, or unknown executables
  • Regular network intervals near 180 seconds

Eradication and recovery

  1. Remove the installer, extracted application, persistence entries, machine identifiers, and second-stage payloads.
  2. Reimage systems when payload scope cannot be established with confidence.
  3. Restore from a verified clean state.
  4. Confirm that beaconing and persistence do not return.
  5. Review the sender mailbox and trusted infrastructure for additional malicious messages.

Analyst Conclusion

PhantomEnigma is not important simply because it used phishing, Node.js, or rotating domains. It is important because it demonstrates how an attacker can combine trusted infrastructure and ordinary software frameworks to make malicious activity look normal at every individual stage.

A legitimate domain can be compromised. An authenticated email can still be malicious. An Electron application can become a backdoor host. A clean automated verdict can hide a malicious build chain. A new C2 domain can make yesterday's blocklist obsolete.

Defenders need detections that connect these events:

Trust anomaly
   + unusual installer chain
   + Electron child process
   + persistence
   + periodic beaconing
   = high-confidence investigation

The most durable indicator in this campaign was not a single hash, IP address, or domain. It was the attacker's engineering pattern.

Final takeaway

Infrastructure tells us where the attacker operated yesterday. Behavior and build-chain evidence help us detect how the attacker will operate tomorrow.
